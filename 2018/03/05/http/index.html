HTTP协议

Http协议通信过程

TCP/IP

http协议属于TCP/IP协议族的子集

TCP/IP的分层管理

应用层(DNS, HTTP), 传输层(TCP协议, UDP协议), 网络层(IP协议), 链路层(硬件相关)

每一次的http请求都会依次经过这四层, 客户端是从应用层到链路层(并且会在请求报文首部添加信息)，服务器端是从链路层到应用层(会依次解封)

IP协议(网际协议)

客户端与服务器端之前需要多台中转设备, IP协议会通过IP地址，使用ARP协议查询路由器的MAC地址，中转数据，层层传递

TCP协议

TCP协议提供可靠的字节流服务

什么是字节流?

把请求报文或者响应报文分割成报文端，称为字节流

什么是可靠的?

TCP的三次握手策略

发送端 –标记SYN的数据包–> 接受端

发送端 <–标记SYN/ACK的数据包– 接受端, 服务器回传

发送端 –标记ACK的数据包–> 接受端, 握手结束

如果握手过程中失败，TCP协议会以相同的顺序发送数据包

DNS服务

位于应用层，提供了域名查找IP地址的解析服务，或者逆向的从IP地址反域名查找

域名 <--> IP地址

HTTP协议

HTTP协议属于无状态协议

不会记录请求和响应

HTTP的方法

  • GET 获取响应主体
  • POST 传输实体的主体
  • PUT 传输文件
  • DELETE 删除制定资源
    — HEAD 与GET类似，但是只返回响应报文的首部
  • OPTIONS 查询支持的HTTP方法
  • TRACE 追踪路径
    — CONNECT 要求建立隧道

持久化链接

在HTTP1.0时代，每一次的http请求前，就要断开TCP链接。

持久链接，TCP链接建立后，在没有明确断开链接的前，会一直保持TCP链接状态（http请求不会经过三次握手）直到TCP链接中断

HTTP1.1默认所有链接都是持久链接

Cookie

Cookie会根据服务端发送的响应报文内Set—Cookie的首部信息字段，通知客户端保存cookie。下一次再次发起请求报文的时候，会自动在报文中加入cookie值发送出去。

客户端 —-> 服务器

客户端 <–响应报文：Set-Cookie– 服务器

客户端 –请求报文：Cookie–> 服务器

HTTP状态码

  • 1XX 信息性状态码
  • 2XX 成功状态码
  • 3XX 重定向状态码
  • 4XX 客户端错误
  • 5XX 服务端错误

常见的状态码

  • 200 请求正常处理
  • 204 请求正常处理，没有资源返回
  • 206 客户端范围请求成功
  • 301 永久性重定向
  • 302, 303 临时性重定向(浏览器会使用GET请求返回Location的URI进行重新请求)
  • 304 使用客户端未过期的缓存
  • 307 与302，303类似，但是不会重新请求
  • 400 请求报文存在语法错误
  • 401 需要认证
  • 403 请求被拒绝
  • 404 服务器无法找到资源
  • 500 服务端错误
  • 503 服务器停机或者超负荷
